# Todo 로컬 웹앱 개발기

> Claude Code로 todomate 대체 앱을 만든 과정 — 기획부터 QA까지

---

## 왜 만들게 되었나

취준, 데놀 회의, e4i2 사이드프로젝트, 비행기 예약, 당근 판매까지 — 해야 할 일이 한꺼번에 쏟아지는 시기였다. 기존에 todomate 앱으로 할일을 관리하고 있었는데, 문제가 있었다:

1. **Claude와 할일이 분리되어 있었다** — Claude에서 면접 준비를 하면서도, 할일 체크는 todomate에서 따로 해야 했다
2. **카테고리가 flat했다** — "취준" 카테고리 하나에 펜타시큐리티, PTK코리아 할일이 섞여서 매번 기업명을 적어야 했다
3. **데이터 접근이 안 됐다** — todomate 데이터를 Claude에서 읽거나 수정할 수 없었다

그래서 Claude Code 안에서 직접 쓸 수 있는 로컬 웹앱을 만들기로 했다.

## Step 1: 기획 — todomate 분석과 요구사항 정리

todomate 스크린샷을 Claude에게 보여주며 "이런 걸 로컬 웹사이트로 만들어줘"라고 요청했다.

**todomate에서 가져올 기능:**
- 카테고리별 할일 그룹 (취준 💰, To-do ✨, 데이터야놀자 📈)
- 할일 체크 (둥근 원형 토글)
- 소요시간 배지 (15m, 1h)
- 예정 시간 표시 (PM 6:00)
- 월간 캘린더 뷰

**기술 결정:**
- 단일 HTML 파일 (외부 의존성 없음)
- Vanilla JS + CSS
- localStorage 저장 (처음에는)
- 브라우저에서 `open todo-app.html`로 실행

## Step 3: 서브카테고리 추가

"취준 카테고리에 기업별 서브카테고리를 나눌 수 있을까?" 라는 요청.

**이유:** 펜타시큐리티와 PTK코리아를 동시에 준비할 때, 매번 "펜타시큐리티 면접 준비"처럼 기업명을 쓰는 게 번거로움. 서브카테고리로 나누면 "면접 준비"만 적어도 어디 기업인지 바로 알 수 있다.

**구현 내용:**
- Category 데이터 모델에 `parentId` 필드 추가 (null = 최상위, 값 = 하위)
- 서브카테고리 UI: 부모 카드 안에 들여쓰기된 소 헤더로 표시
- 서브카테고리 추가/이름변경/삭제 기능
- 서브카테고리 없는 카테고리는 기존처럼 flat 동작 (하위 호환)

**초기 데이터 변경:**
```
Before: 취준 > 펜타시큐리티 면접 준비
After:  취준 > 펜타시큐리티 > 면접 준비
```

## Step 4: JSON 파일 저장 (localStorage → 파일)

"DB에도 저장할 수 있어?" 라는 질문에 3가지 옵션을 제시:
1. JSON 파일 저장 (Node.js 서버)
2. SQLite
3. Notion DB 동기화

**1번(JSON 파일) 선택 이유:**
- 별도 DB 설치 불필요
- Claude에서 `todo-data.json` 파일을 직접 읽고 수정 가능
- 브라우저 캐시 삭제해도 데이터 안 날아감

**구현:**
- `todo-server.js` — Node.js HTTP 서버 (API + HTML 서빙)
- `GET /api/data` — JSON 파일 읽기
- `POST /api/data` — JSON 파일 쓰기
- 프론트엔드: API 우선, 실패 시 localStorage fallback
- 양쪽 모두에 dual write (백업)

## Step 5: 버그 사냥 — POST가 안 나가는 미스터리

JSON 파일 저장을 구현한 후, 브라우저에서 `todo-data.json`이 생성되지 않는 문제가 발생했다. 서버 로그에 `GET /api/data`만 찍히고 `POST /api/data`는 한 번도 호출되지 않았다.

### 디버깅 과정

**1차 시도: `location.origin` 확인**
- `file://` 프로토콜에서 `location.origin`이 `null`이 될 수 있어서, API URL을 `http://localhost:8080`으로 하드코딩
- 결과: 여전히 POST 안 나감

**2차 시도: async/await 문제 의심**
- `loadData()`는 `async`인데 `saveData()`는 일반 함수
- `await saveData()` 호출이 실제로는 동기적으로 실행되어 fetch가 취소될 수 있다고 판단
- `saveDataAsync()` 함수를 별도 추가하고 첫 로드 시에만 사용
- 결과: 여전히 POST 안 나감

**3차 시도: 서버 쿼리스트링 처리 누락 발견**
- 캐시 우회를 위해 `?t=timestamp`를 붙여 열었는데, 서버가 `req.url === '/'`로만 매칭해서 404 반환
- `req.url.split('?')[0]`으로 수정
- 결과: HTML은 서빙되지만 여전히 POST 안 나감
- (이때 유저는 "Not Found 뜨는데"라고 알려줌 — 이게 실마리였음)

**4차 시도: 서버 요청 로깅 추가**
- `console.log(\`${req.method} ${req.url}\`)` 추가해서 실제 요청 흐름 추적
- `GET /`만 찍히고 `GET /api/data`도 없었음 → JS가 아예 실행 안 되고 있었음
- 브라우저 캐시가 옛날 HTML(file:// 시절)을 로드하고 있었던 것

**5차 시도: QA 에이전트 투입 — 근본 원인 발견**
- 체계적 QA를 돌려보니 `saveData` 함수가 **2번 선언**되어 있었음
- 1번째 (line 1022): API + localStorage 저장 (올바른 버전)
- 2번째 (line 1062): localStorage만 저장 (잘못된 버전)
- JavaScript에서 같은 이름의 function 선언은 마지막이 이김 → API 저장 코드가 덮어씌워짐

**근본 원인:** 서브카테고리 기능을 추가할 때 코드 블록을 이동하면서, 기존 `saveData`가 남아있었고 새로운 `saveData`(API 버전)가 추가됨. 결과적으로 2개가 공존하게 되었고, JavaScript hoisting으로 인해 뒤쪽 선언(localStorage만)이 우선함.

**해결:** 중복된 `saveData` 선언 삭제. 1줄 수정으로 해결.

### 교훈
- function 중복 선언은 에러를 던지지 않아 발견하기 어렵다
- 서버 로깅은 디버깅의 기본인데, 처음부터 넣었어야 했다
- 체계적 QA(자동화된 검증)가 수동 디버깅보다 빠르게 근본 원인을 찾았다

## Step 8: 드래그 앤 드롭 — 카테고리 간 할일 이동

"다른 카테고리에 쓴 할일을 드래그해서 옮길 수 있을까?" 라는 요청.

HTML5 Drag & Drop API로 구현:
- todo 항목에 `draggable="true"` 속성 추가
- 각 카테고리/서브카테고리 body에 `data-drop-cat` 속성으로 드롭 영역 지정
- 드래그 시 반투명, 드롭 영역에 올리면 배경색 변화로 시각적 피드백
- 드롭하면 `todo.categoryId` 변경 + 자동 저장

서브카테고리가 있는 카테고리에서는 각 서브카테고리별 영역이 독립적으로 드롭존 역할을 한다.

## Step 9: "내일하기" 기능

할일 ··· 메뉴에 "📅 내일하기" 추가. 누르면 해당 할일의 날짜가 내일로 이동하고, 완료 상태는 초기화된다. 소요시간, 예정시간 등 세팅값은 그대로 유지.

구현은 단순:
```js
const d = new Date(todo.date);
d.setDate(d.getDate() + 1);
todo.date = toDateStr(...);
todo.done = false;
```

## Step 10: 미룬 일수 추적 (D+N 배지)

"내일하기"로 계속 미루면 원래 언제 만든 건지 알 수 없었다. 처음 만든 날로부터 며칠째 미루고 있는지 알고 싶다는 요청.

**데이터 구조 변경:**
- todo에 `createdDate` 필드 추가 (최초 생성 날짜, 변경 안 됨)
- 기존 데이터 마이그레이션: `createdDate` 없으면 `date` 값으로 자동 세팅

**표시:**
- `date - createdDate > 0`이면 빨간 **D+1**, **D+2** 배지 표시
- 완료된 항목에는 미표시
- 당일 생성 항목에는 미표시

이 기능을 만들면서 느낀 건, 데이터 구조 변경이 생길 때 마이그레이션 패턴이 중요하다는 것. loadData()에서 필드 유무를 체크하고 없으면 기본값을 세팅하는 방식으로, 기존 사용자 데이터를 깨뜨리지 않고 자연스럽게 전환할 수 있었다.

## Step 11: 카테고리 수정/삭제 + 서브카테고리 관리 개선

여러 요청이 연속으로 들어왔다:

**카테고리 수정/삭제:**
- 카테고리 헤더에 ··· 버튼 추가 (hover 시 표시)
- 수정: 기존 카테고리 추가 모달을 재활용 (`editingCatId` 플래그로 추가/수정 분기)
- 삭제: 하위 서브카테고리 + 할일 전부 삭제 (confirm 확인)

**서브카테고리 추가 방식 변경:**
- 처음: 서브카테고리가 이미 있는 카테고리에서만 인라인 바로 추가 가능
- 문제: 서브카테고리가 없는 카테고리에서는 추가할 방법이 없음
- 해결: 카테고리 ··· 메뉴에 "📁 서브카테고리 추가" 옵션 추가 → 인라인 바 제거

이렇게 하면 모든 카테고리에서 서브카테고리를 만들 수 있으면서도, UI가 깔끔하게 유지된다.

## Step 12: 서브카테고리 숨기기/복원 (아카이브)

"펜타시큐리티 준비가 끝나면 카테고리가 계속 보이는데, 데이터는 보관하면서 안 보이게 할 수 없을까?"

**핵심 고민:**
- 단순 숨기기는 쉬운데, 과거 날짜에 해당 서브카테고리로 작성한 할일이 있으면 그 날에는 보여야 한다
- 숨긴 걸 다시 보이게 하는 UX도 필요

**구현:**
- category에 `archived` 필드 추가
- 서브카테고리 ··· 메뉴에 "📦 숨기기"
- 렌더링 로직: `archived && 이 날짜에 할일 0개 → 스킵`, `archived && 할일 있음 → 반투명 + (숨김) 라벨`
- 부모 카테고리 ··· 메뉴를 동적으로 생성해서, 숨긴 서브카테고리 목록을 표시하고 클릭하면 복원

**복원 UX:** 부모(취준) ··· 메뉴 하단에 구분선 + "숨긴 서브카테고리: ↩️ 펜타시큐리티" 형태로 표시.

## Step 14: 로고 반영 + 기한 만료 자동 처리

dutodo 로고(v3)를 디자인하고 앱에 반영. 두두 캐릭터(체크마크 귀 + 하얀 얼굴)를 SVG로 제작.

**로고 반영:**
- 브라우저 탭 파비콘 (inline SVG data URI)
- 좌측 패널 상단 워드마크 (아이콘 + "dutodo" + "do it, dudu!" 슬로건)

**기한 만료 자동 처리:**
- 할일의 날짜가 오늘보다 이전 + 미완료 → 자동으로 회색 + 취소선 + 투명도 45%
- D+N 배지도 기한 만료 항목에는 미표시 (이미 지난 거니까)
- "내일하기"로 다시 살릴 수 있음 (기한 제한 없이 유지)

"포기" 체크 버튼을 넣을까 고민했으나, 매번 "포기"를 직접 누르는 건 심리적으로 부정적 경험이 되므로 자동 시각 처리로 결정.

## Step 15: 카테고리 드래그 순서 변경

카테고리 카드를 드래그해서 다른 카테고리 위로 옮기면 순서가 변경되도록 구현.

- HTML5 Drag & Drop API 활용 (할일 드래그와 별도 데이터 타입 `application/cat-id`로 분리)

할일 드래그(`text/plain`)와 카테고리 드래그(`application/cat-id`)가 충돌하지 않도록 dataTransfer 타입을 분리한 것이 핵심.

## Step 16: 타임라인 뷰

"카테고리별로 보면 시간순 하루 흐름이 안 보인다"는 문제 인식에서 시작.

**구현:**
- 좌측 패널 "오늘 완료" 아래에 시간순 미니 타임라인 추가
- 시간 설정된 항목은 시간순 정렬, 미설정 항목은 "시간 미정"으로 하단 분리
- 카테고리 색상 체크 버튼 + 카테고리명 + 소요시간 표시
- 체크 클릭으로 완료/미완료 토글 (오른쪽 카테고리 뷰와 동기화)

## Step 18: 실제 소요시간 추적 + 타임테이블 뷰

todomate 대체로 쓰다가 생긴 두 가지 불편함에서 시작.

**문제 1:** 예상 15분으로 잡은 일이 실제로는 2시간 넘게 걸리는 경우가 잦은데, 예상 대비 얼마나 밀렸는지 흔적이 안 남음. "15분이라 적어놨는데 한참 걸렸네" 정도만 기억.

**문제 2:** 하나가 밀리면 뒤에 시간 세팅해둔 할일들을 전부 수동으로 옮겨야 해서, 사실상 시간 세팅을 포기하게 됨. 구글캘린더처럼 시간축 위에서 직접 블록을 드래그/리사이즈하고 싶음.

### 데이터 모델 개편

- `duration` → `estimatedTime`으로 리네임, `actualTime` 필드 신규 추가
- `loadData()` 마이그레이션: 기존 `duration` 값은 자동으로 `estimatedTime`으로 이관 후 제거
- 포맷 헬퍼 추가: `parseDuration('1h 30m') → 90`, `formatDuration(105) → '1h 45m'`
- 15분 단위 프리셋: `15m / 30m / 45m / 1h / 1h 30m / 2h / 2h 30m / 3h`

### 리스트 뷰 인라인 표시

`⏱ 15m → 2h +1h 45m` 형식으로 예상과 실제를 함께 보여주고, 초과분은 빨간 뱃지, 단축은 녹색 뱃지로 시각화. 예상시간만 있거나 실제시간만 있는 케이스도 각각 처리.

### 완료 시 실제시간 입력 UX

체크박스 클릭 시, 예상시간이 있는데 실제시간이 비어 있으면 하단에 토스트가 뜬다. 프리셋 칩(`예상대로 / 15m / 30m / 1h / 1h 30m / 2h`) + 직접 입력 + "나중에" 스킵. 강제하지 않는 것이 핵심 — 기록은 많이 남길수록 좋지만 매번 귀찮게 만들면 앱을 안 쓰게 된다.

### 수정 모달 — 칩 리스트로 전환

처음엔 `<select>` 드롭다운으로 구현했는데, select 옵션 라벨이 "15분/1시간"(한글)이고 사용자가 직접 입력한 실제시간은 내부 포맷(`2h`)이라 섞여서 들쭉날쭉해 보였음. 가로 스크롤 가능한 **칩 리스트**로 교체:

```
[없음] [15분] [30분] [45분] [1시간] [1시간 30분] [2시간] [2시간 30분] [3시간]
```

- 클릭으로 선택
- 공간 부족하면 가로 스크롤
- 리사이즈로 생긴 비표준 값(`1h 45m` 등)은 끝에 **점선 테두리 칩**으로 자동 추가 — 데이터 손실 없이 재선택 가능

### 타임테이블 뷰 v1

오른쪽 패널 상단에 `[📋 리스트 | 📅 타임테이블]` 토글 버튼 추가. localStorage로 선택 상태 유지. 같은 할일 데이터를 공유하되 표현 방식만 바꿈.

**레이아웃:**
- 00~24시 세로 스크롤, 1시간 = 72px
- 30분 슬롯 48개로 그리드 구성 (처음엔 15분 단위로 쪼갰는데 너무 세밀해서 30분으로 축소, 15분짜리는 한 칸의 절반으로 표시)
- 정각에 시간 라벨 (`0 1 2 ... 23`) — 초기 구현 시 `padding-left + absolute child` 조합으로 라벨이 음수 좌표에 떨어져서 잘리는 이슈가 있었고, 패딩 제거 + offset 기반 재설계로 해결
- 오늘이면 빨간 현재시각 라인 + 초기 스크롤 자동 중앙

**블록 렌더링:**
- 예상시간 = 점선 아웃라인 (카테고리 색)
- 실제시간 = 파스텔 채움 (카테고리 색, opacity 0.28)
- 예상 초과분 = 빨강 채움 — 예상 박스 밖으로 빨간 영역이 삐져나오는 형태라 한눈에 "얼마나 밀렸는지"가 보임
- 완료 시 취소선 + 투명도
- 겹치는 블록은 나란히(side-by-side) 배치. 그리디 컬럼 알고리즘으로 그룹화

**인터랙션:**
- 드래그로 블록 이동 → `scheduledTime` 변경 (15분 스냅)
- 위/아래 가장자리 리사이즈
  - **규칙**: `estimatedTime`이 없으면 → 생성, 있으면 → `actualTime` 기록
  - 이유: 처음 만들 때는 예상시간을 세팅하는 용도, 실제 일을 마친 후에 다시 리사이즈할 일이 더 많을 테니 그때는 실제시간으로 자동 저장
- `scheduledTime`만 있고 예상시간 없는 할일은 30분 점선 블록으로 기본 표시
- 시간 미정 할일은 하단 칩 섹션 → 드래그해서 시간축에 드롭하면 그 위치로 배치
- 블록 클릭(드래그 아님) → 수정 모달

**결정하지 않은 것들 (v2):**
- 자정 넘김: 일단 24:00에서 컷 + 토스트 (열두시 넘어서 작업하는 경우 많지만 v1 단순화)
- 자동 시프트(하나 밀리면 뒤 일정 자동 이동)
- 주간 뷰
- 타이머 버튼 (실제시간 자동 측정)

### 기획 → 구현 과정에서 얻은 것

이번 작업은 유난히 대화 초반에 "예외 상황 / 더 고려할 지점"을 미리 열거하는 데 시간을 많이 썼다. 겹침 처리, 리사이즈가 바꾸는 값, 미정 할일, 자정 넘김, 시간 범위, 밀린 할일 — 이 중 사용자가 직접 결정한 항목은 모두 기록에 남기고 구현에 반영했다. 덕분에 중간에 "어? 이건 어떻게 하지?"로 멈추는 일 없이 단계별로 커밋할 수 있는 수준까지 정리됐다.
